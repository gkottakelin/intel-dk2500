# 摘要

本作品面向桌面物块抓取、放置与叠放，设计云端/本地模型可切换的RGB视觉闭环机械臂。系统以600×480、60 fps图像为输入，由模型解析自然语言，本地完成目标与夹爪检测、任务校验、运动学求解和舵机控制。抓取采用图像误差迭代与J6位置稳定判定，放置对目标对准、下降、释放及结果确认持续闭环。设计指标为检测准确率大于95%，夹爪及TCP定位误差小于2 cm，闭环收敛小于3 s，单次任务小于1 min。

关键词：机械臂；自然语言控制；本地模型；RGB视觉；视觉闭环；目标检测

# Abstract

This work presents a desktop robotic arm system for object grasping, placement, and stacking using switchable cloud and local models together with RGB visual closed-loop control. The system uses 600 × 480 color images at 60 frames per second as its perception input. A selected model backend converts natural-language instructions into a structured task, while the local controller performs object and gripper detection, schema and safety validation, kinematic computation, bus-servo control, and fault handling. During grasping, image-space errors are iteratively reduced and gripper contact is inferred from the stability of the J6 position feedback. During placement, visual feedback is retained through target alignment, incremental descent, gripper release, and post-release verification. The design targets are an object-detection accuracy above 95%, gripper and TCP localization errors below 2 cm, visual-loop convergence within 3 s, and completion of one task within 1 min. Parameters without hardware evidence are explicitly treated as acceptance targets rather than measured results.

Keywords: Robotic Arm; Natural-Language Control; Local Model; RGB Vision; Visual Closed-Loop Control; Object Detection

# 第一章 绪论

## 1.1 项目背景

桌面机械臂兼具结构紧凑、成本可控和实验安全边界清晰等特点，是机器人运动学、机器视觉、智能交互与闭环控制融合验证的重要平台。传统桌面机械臂通常依赖固定动作脚本、示教点或专用示教器。当目标位置、颜色或放置区域发生变化时，操作者需要重新示教，系统对环境变化的适应能力有限。若直接把自然语言、大模型和机械臂底层执行连接起来，又会引入输出不确定、动作不可验证以及网络依赖等风险。因此，一个可用的智能机械臂系统不能只解决“理解指令”，还必须建立明确的本地安全边界和可观测的执行闭环。

本作品以JetArm总线舵机机械臂为执行平台，以安装在机械臂末端附近的Gemini Pro Plus相机为视觉传感器，以Intel DK2500或具备等效算力的上位机为运行载体。用户可以通过自然语言描述“夹取红色物块”“将红色物块放入蓝色区域”或“将一个物块叠放到另一个物块上”等任务。系统把语言理解、目标选择、视觉定位、运动规划、夹取、放置和结果确认分解为边界清晰的模块，并以统一的任务JSON连接模型与本地执行器。

作品采用RGB图像作为主方案输入。与仅依赖开环关节位置的方案相比，RGB视觉闭环在每次小步动作后重新观察目标和夹爪，通过图像误差修正机械臂运动，可以降低舵机回差、装配误差、负载变形和目标位置变化对最终结果的影响。系统同时保留Gemini相机深度实验代码和固定尺寸单目测距方案，作为后续扩展，而不把未经集成的深度功能写入主方案的已实现能力。

## 1.2 研究目的与意义

本项目的直接目标是构建一套能够接受自然语言任务、识别桌面目标并通过视觉反馈完成抓放操作的机械臂原型。其工程意义体现在三个方面。第一，通过云端模型与本地模型可切换的设计，系统既能利用云端多模态模型的复杂场景理解能力，也能在网络不可用、隐私要求较高或需要稳定响应时间时选择本地模型。第二，通过统一JSON Schema、动作白名单和本地安全检查，把不确定的模型输出约束为有限、可验证的任务步骤，使模型不接触舵机原始位置、速度或串口命令。第三，通过抓取和放置全过程闭环，把“视觉确认”从任务末尾的附加步骤提升为控制回路的一部分。

对于教学和竞赛场景，本作品覆盖总线通信、关节标定、正运动学、数值雅可比、阻尼最小二乘、RGB目标检测、视觉伺服、状态机和人机界面等多个知识点。系统的软件层次和接口具有较强可复用性，后续可以替换目标检测器、模型后端或相机，而不必重写舵机底层控制。

## 1.3 主要研究内容

本作品围绕以下内容展开：一是依据总线舵机协议实现半双工UART通信，完成数据帧组装、校验、位置控制、速度控制以及温度、电压和位置反馈读取；二是根据机械臂实测尺寸建立J1至J5串联运动学模型，完成舵机原始位置、关节角和TCP位置之间的转换；三是开发Windows和Ubuntu 22.04操作终端，实现J1至J4笛卡尔小步控制、J5腕部旋转、J6夹爪开合、Home和全部停止；四是构建云端/本地模型统一接口，使用结构化任务和动作白名单隔离模型与底层硬件；五是设计目标检测、夹爪标记检测、抓取闭环、放置闭环以及失败恢复流程；六是建立覆盖软件单元测试、视觉性能、定位误差、闭环时间和整机任务耗时的测试方法。

## 1.4 技术难点

眼在手上的相机会随J1至J5运动，使同一个目标在图像中的运动方向和尺度随机械臂姿态变化，固定的“像素向左对应J1增加”规则只能在局部区域成立。项目因此需要在工作姿态附近标定视觉雅可比，或在运行中通过小步扰动更新像素误差与关节增量之间的关系。另一个难点是J2承担较大负载，接近水平姿态时实际位置误差可能增大，必须使用位置反馈和视觉反馈共同修正，而不能把上一条控制命令当作真实关节状态。

RGB图像缺乏直接深度测量，放置高度尤其是叠放接触高度不能仅凭单帧中心点确定。本方案采用小步下降、多帧目标尺度和遮挡变化分析、任务几何约束及释放后验证组合判定。模型输出的不确定性则通过JSON Schema、对象ID引用、动作白名单、关节限位和桌面高度约束控制在本地执行器之外。

## 1.5 报告结构

第二章给出任务场景、功能需求、性能指标和约束；第三章说明系统总体方案、模型后端和闭环链路；第四章介绍硬件组成、关节参数、通信和连接关系；第五章推导舵机控制、运动学、目标检测及视觉闭环原理；第六章说明软件模块、状态机和异常处理；第七章给出测试方案、设备、已获得的软件测试数据及待完成的真机数据表；第八章总结作品已实现功能、技术特色、局限和后续工作。附录列出程序、核心代码、系统连接资料和参考文献。

# 第二章 系统需求与性能指标

## 2.1 典型任务场景

系统面向桌面范围内的彩色物块、放置框和叠放目标。用户不需要输入舵机ID、关节角或坐标，只需描述目标属性和动作关系。单目标抓取任务要求系统选择符合颜色或类别描述的物体，对准夹爪并夹持；抓取放置任务要求系统在抓取后找到指定区域，完成对准、下降、释放和结果确认；叠放任务除目标选择外，还要求判断被抓物体与承载物体的相对位置，并在释放后确认两者形成稳定叠放关系。

系统启动后首先进行设备与配置检查，随后回到Home姿态。每个任务均从当前RGB图像和真实舵机状态出发，而不是复用上一次任务的目标位置。若目标、夹爪或放置区域不可见，系统不得盲目运动，应停止并请求重新观察或人工处理。

## 2.2 功能需求

### 2.2.1 自然语言任务理解

系统应能把用户语言转换为任务类型、源目标、目的目标、动作步骤和成功条件。例如“把红色物块放进蓝色框”应被解析为pick_place任务，并引用当前视觉状态中真实存在的对象ID。若用户描述不完整或场景中存在多个同类对象，模型应返回需要重新观察或澄清的结果，不能任意选择。

### 2.2.2 云端/本地双模型

模型后端支持cloud、local和auto三种配置。cloud模式调用云端多模态模型；local模式调用部署在上位机的本地模型；auto模式在任务开始时优先尝试配置的主后端，调用失败或超时后切换备用后端。两种后端必须遵循相同的输入输出接口。本地程序只接收结构化任务，不接受模型生成的串口命令、关节原始值或任意代码。为避免同一任务执行逻辑中途改变，后端选择在任务开始阶段锁定，任务执行过程中不静默切换。

### 2.2.3 RGB图像采集与视觉检测

系统设计输入为600×480、60 fps RGB彩色图像。视觉模块应输出图像时间戳、目标框、目标中心、置信度、左右爪尖、夹爪中心、夹爪开口方向和可见性。复杂物体类别可由模型后端辅助识别，局部实时闭环则优先使用低延迟的本地检测结果。相机实际是否支持该分辨率与帧率组合，需要在真机测试中通过连续采集和丢帧统计确认。

### 2.2.4 机械臂执行与状态反馈

J1至J4采用位置模式执行机械臂主体运动，J5采用速度模式调整腕部旋转，J6采用速度模式完成夹取与释放。控制器应能读取舵机位置、温度和输入电压。Home参考值为J1=500、J2=550、J3=550、J4=900、J5=500、J6=360；其中J6物理完全张开位置为0，360是当前任务流程采用的松开参考位置，最终释放终点应在真机测试后统一配置。

### 2.2.5 抓取闭环

抓取前，系统反复检测目标与夹爪，计算目标中心相对两爪中心线的误差，执行受限小步动作并重新拍摄。只有目标进入两爪之间、定位误差满足阈值并连续多帧稳定时，才允许J6闭合。J6以较低速度接近物体，持续读取位置；若位置在规定容差内连续2 s基本不变，则判定夹爪与物体接触，再提高夹紧速度。抬起后重新检测目标是否随夹爪移动，防止把空夹或滑落误判为成功。

### 2.2.6 放置闭环

放置过程同样保持闭环。系统首先检测放置目标、夹爪和被夹物体，计算水平误差、姿态误差和释放高度条件；未满足条件时，J1至J4小步修正并逐步下降，每一步后重新观察。满足释放条件后，J6以负速度张开并根据位置反馈停止。释放后系统再次采集RGB图像，只有被夹物体已经离开夹爪、位于目标区域且连续多帧稳定，才判定任务成功；若物体仍粘连、明显偏移或倾倒，则根据当前可恢复性重新调整、重新抓取或安全终止。

## 2.4 安全与约束要求

安全控制采用“模型输出约束、运动规划约束和执行反馈约束”三层结构。模型层只能从动作白名单中选择步骤；规划层检查对象ID、关节范围、桌面高度和单步幅度；执行层监测串口响应、舵机位置和视觉状态。J2为高负载关节，禁止使用Motor模式。J5、J6使用速度模式时必须具备松手停止、全部停止和超时退出。目标或夹爪丢失、视觉闭环长期不收敛、舵机反馈异常、模型输出非法或任务超时均触发abort。任务正常完成后统一回Home；异常时仅在回Home路径可确认安全的情况下自动返回，否则保持停止并等待人工处理。

# 第三章 系统总体方案

## 3.1 设计原则

系统遵循四项原则。第一，模型负责理解和高层决策，本地程序负责校验和执行。任何模型都不能绕过安全控制直接访问总线舵机。第二，局部控制依赖实时视觉，不要求云端模型逐帧返回，避免网络延迟破坏闭环稳定性。第三，机械臂每次只执行受限小步动作，并在动作后重新读取状态、重新拍图。第四，抓取和放置都以可观测成功条件结束，而不是以“命令已经发送”作为成功判据。

## 3.2 系统总体架构

系统由交互层、模型层、视觉层、任务规划层、闭环控制层、执行层和安全层组成。交互层接收自然语言和人工控制；模型层根据用户指令、RGB图像、视觉状态和机器人状态输出结构化任务；视觉层负责高频图像采集、目标与夹爪检测；任务规划层把结构化任务映射为观察、对准、夹取、抬升、放置、释放、验证和回Home等状态；闭环控制层计算图像误差或TCP增量；执行层将动作转换为舵机协议帧；安全层横跨所有模块并拥有停止权限。

## 3.3 云端/本地模型选择方案

模型管理器对上层暴露统一的parse_task接口，输入包含user_request、当前RGB图像、robot_state、vision_state和allowed_actions。云端后端适用于复杂场景语义理解和开放类别目标选择；本地后端适用于离线运行、低网络依赖和稳定隐私边界。系统不规定唯一模型型号，部署时应根据Intel DK2500或上位机的CPU、GPU/NPU能力选择能够在目标延迟内运行的模型。

auto模式不是简单地在每帧图像上竞争调用两个模型。系统先调用主后端，若发生连接失败、超时或返回格式错误，则在机械臂尚未启动动作时调用备用后端。备用结果仍需通过同样的Schema和安全检查。若任务已经进入执行阶段，模型失效不触发无条件切换，而是由任务状态机决定继续使用已验证计划、重新观察或中止。

## 3.4 结构化任务与动作白名单

模型输出至少包含task_id、task_type、target、destination、steps和risk_checks。target与destination必须引用vision_state中存在的object_id，不能只返回模糊文本。允许的动作包括observe、select_target、align_to_target、grasp、lift、align_to_destination、release、verify、return_home和abort。对齐动作只表达意图，不携带关节位置；具体关节增量由本地闭环控制器计算。

典型任务JSON如下：

## 3.6 软件总体流程

启动时，程序加载舵机ID、限位、Home、机械尺寸、相机和模型后端配置，发现串口并读取关节状态；自检通过后回Home并进入IDLE。收到任务后采集RGB图像并生成视觉状态，模型返回结构化任务，本地完成Schema、对象ID、动作白名单和风险检查。执行抓取时先闭环对准，再完成J6接触检测与抬起验证；执行放置时保持对准、下降、释放位置和释放结果四个环节的反馈。任一环节失败均进入受控恢复或ABORT。

## 3.7 全过程闭环设计

本系统把一个完整任务划分为多个闭环，而不是只在抓取前进行一次视觉定位。第一层是抓取对准闭环，目标中心与夹爪中心误差驱动J1至J4小步动作；第二层是夹取接触闭环，J6位置变化用于判断是否接触；第三层是抓取结果闭环，抬升后确认物体随夹爪移动；第四层是放置对准与下降闭环，每一步重新计算目标、夹爪和被夹物体的相对位置；第五层是释放位置闭环，J6根据位置反馈停止；第六层是释放结果闭环，通过多帧图像判断物体是否离开夹爪并稳定落在目标区域。该结构使任务成功与真实环境状态绑定。

# 第五章 系统实现原理

## 5.1 总线舵机通信协议

舵机通信帧由双字节帧头0x55 0x55、舵机ID、长度、指令、参数和校验和组成。程序从ID字段开始累加到参数区末尾，对结果按位取反并保留低8位：

checksum = (~sum(ID, Length, Command, Parameters)) & 0xFF

发送位置命令时，参数包括目标位置和运行时间，均按小端序16位整数编码；发送Motor命令时，参数包括模式、转动方式和有符号速度。读取温度、位置和电压时，控制器发送读取指令并在超时范围内寻找0x55 0x55帧头，再检查返回ID、命令、长度和校验和。任何一项不一致均抛出异常，避免错误数据进入规划层。

[FIGURE: 图5-1 总线舵机协议帧结构 | 内容: 按字节展示帧头、ID、长度、命令、参数和校验和，并标明位置与Motor命令的参数格式 | 建议来源: 根据《02 总线舵机通信协议》绘制]

## 5.2 舵机位置与关节角映射

各关节原始位置和模型角度采用分段范围内的线性映射。对给定原始位置p，其舵机角度为：

θservo = θmin + (p - pmin) / (pmax - pmin) × (θmax - θmin)

再乘以方向符号得到模型角度θmodel。J1和J5的配置方向符号为负，用于使舵机实际旋转方向与右手坐标系一致。反向计算时先除以方向符号，再按比例映射回原始位置并取整。映射前后均检查关节范围，超出配置边界的目标直接拒绝。

J2存在负载相关误差，接近90°姿态时配置记录约10个原始位置单位的偏差。该误差不通过任意固定补偿完全消除，而是通过读取真实位置、视觉闭环和后续实测标定共同处理。

## 5.3 正运动学模型

设J1模型角为q1，J2至J4模型角为q2、q3、q4，J5滚转角为q5。J2至TCP在未施加q1偏航前位于X-Z平面。定义h为基座至J2高度，L23、L34、L45和L5T为相邻连杆长度，则平面内TCP坐标为：

x = L23 sin(q2) + L34 sin(q2+q3) + (L45+L5T) sin(q2+q3+q4)

z = h + L23 cos(q2) + L34 cos(q2+q3) + (L45+L5T) cos(q2+q3+q4)

基座偏航q1把平面径向坐标旋转到基座X-Y平面：

X = x cos(q1)，Y = x sin(q1)，Z = z

末端姿态由偏航、累计俯仰和J5滚转共同决定。当前简化模型认为J5只改变TCP方向，不改变TCP位置，J6仅表示夹爪状态。模型输出各关节原点、TCP坐标和旋转矩阵，可用于终端运动控制、方案3单目测距坐标转换以及URDF一致性验证。

[FIGURE: 图5-2 机械臂运动学模型 | 内容: J1偏航和J2至J4平面连杆示意，标注q1至q5、h、L23、L34、L45、L5T和TCP | 建议来源: 根据机械尺寸绘制]

## 5.4 数值雅可比与阻尼最小二乘

为了把用户给出的笛卡尔速度或视觉控制产生的TCP增量转换为关节增量，程序使用中央差分计算平移雅可比。对第i个关节施加±δ扰动，分别计算TCP位置p(q+δei)和p(q-δei)，则雅可比第i列近似为：

Ji = [p(q+δei) - p(q-δei)] / (2δ)

直接使用伪逆在接近奇异姿态时会产生过大的关节增量。本项目操作终端采用阻尼最小二乘形式：

Δq = Jᵀ(JJᵀ + λ²I)⁻¹Δx

其中λ为阻尼系数。求得Δq后按每周期最大4°裁剪，再转换为舵机原始位置并应用关节限位。程序还以当前解为中心，对各关节±8个原始单位及组合俯仰方向进行局部搜索，选择使TCP目标误差更小的候选点。若所有候选都不能改善误差，则保持当前位置并提示达到限位或当前姿态无法继续。

## 5.5 RGB目标检测

主方案的近距离彩色目标可使用HSV颜色分割建立低延迟基线。程序将BGR图像转换到HSV空间，对目标颜色使用上下阈值生成二值掩膜，再通过5×5结构元素进行开运算去除孤立噪声、闭运算填补区域空洞。随后提取外轮廓，过滤小面积区域，选择满足面积、长宽比和位置约束的候选，输出包围框、中心点、面积和置信度。

颜色检测适用于颜色明确、背景受控的竞赛场景。复杂类别、同色多目标和自然语言属性组合由云端或本地模型提供候选对象和语义标签，本地检测器负责在后续帧中跟踪已选对象。为满足“目标检测准确率大于95%”的设计指标，正式测试必须覆盖不同光照、位置、遮挡和背景，不能只使用单一演示视频。

## 5.6 夹爪标记检测与定位

黑色夹爪本体容易与阴影和背景混淆，因此系统在两侧爪尖设置橙色标记，在夹爪主体设置紫红色辅助区域。检测器优先对橙色进行HSV分割，在几何约束下选择两个稳定连通域，分别作为left_tip和right_tip。夹爪中心、开口宽度和中心线定义为：

gripper_center = (left_tip + right_tip) / 2

gripper_width = ||right_tip - left_tip||

gripper_axis = normalize(right_tip - left_tip)

若只检测到一个橙色区域，程序利用紫红色主体、上一帧状态和合理开口范围尝试恢复；连续丢失则停止闭环。像素坐标可通过标定平面上的像素—厘米映射或相机模型转换为空间误差。夹爪定位误差小于2 cm是待实测目标，应以标定板或外部测量作为真值计算平均欧氏误差。

## 5.7 视觉雅可比标定

运动学雅可比描述关节角对TCP空间位置的影响，视觉雅可比描述关节小步动作对图像特征误差的影响，两者不能混为一谈。系统在Home附近和典型工作姿态分别进行小步扰动：对J1至J4施加正负原始单位增量，记录动作前后夹爪中心、目标中心和中心线角度变化，构成局部视觉响应矩阵。对图像误差e=[du,dv]ᵀ，可通过伪逆或阻尼最小二乘求得关节修正量：

Δq = -K Jv⁺ e

其中K为小于等于1的增益。运行时限制每个关节的最大步长，并在每步后重新检测。若误差方向连续反转或数值增大，控制器降低增益、重新估计局部响应或中止，防止振荡。

## 5.8 抓取闭环原理

抓取闭环以目标中心与夹爪中心的像素误差、目标是否位于两爪之间以及夹爪姿态误差作为收敛条件。系统从观察姿态开始，检测结果有效时计算误差，经视觉雅可比得到J1至J4小步动作；执行后等待图像稳定并重新检测。误差连续若干帧低于阈值才进入闭合状态，避免单帧噪声触发抓取。

J6以速度100闭合并周期读取位置。位置稳定跟踪器以当前参考位置和参考时间为状态，当新位置偏离参考值超过容差时更新参考；若在容差内保持2 s，则认为夹爪受物体阻挡。随后提高速度至300短暂夹紧并停止。抬升动作后检测目标是否与夹爪同步移动，如果目标仍留在桌面或无法检测到合理的被夹状态，则判定抓取失败。

## 5.9 放置与释放闭环原理

放置阶段检测三个对象：放置目标、夹爪和被夹物体。水平对准误差可以由被夹物体中心与放置目标中心计算，姿态误差由目标主方向与夹爪方向计算。对于平面放置，系统在安全预放置高度完成水平对准，再小步下降；对于叠放，结合被夹物体与承载物体的相对尺度、轮廓接近和遮挡变化判断释放条件。每次下降后必须重新拍图，禁止一次性下降到未经验证的固定高度。

满足释放条件后，J6以速度-100张开并持续读取位置，到达配置的松开位置或超时后发送0速度。释放后重新检测场景：若物体已经离开夹爪、中心位于目标区域、相对位姿合理并连续多帧稳定，则任务成功；若物体仍与夹爪同步、明显偏离目标或倾倒，系统进入恢复判断。可以安全恢复时重新对准或重新抓取，不可恢复时停止并报告失败。

## 5.10 结果判定与安全恢复

结果判定采用多源条件而不是单一置信度。视觉条件包括目标和夹爪可见性、物体中心是否位于目标区域、物体是否离开夹爪以及状态是否连续多帧稳定；舵机条件包括J6位置、关节是否越限、反馈是否及时；任务条件包括总耗时、闭环迭代次数和成功条件。只有全部必要条件满足才转入RETURN_HOME。

恢复策略按照风险递增排序：重新观察、降低步长继续对准、退回安全高度重新规划、重新抓取、停止并等待人工处理。回Home不是所有异常的无条件动作；当相机丢失且无法确认路径、舵机反馈异常或机械臂可能与环境接触时，立即停止比自动回Home更安全。

## 6.3 总线控制软件

bus_servo.py提供BusServoController上下文管理器，支持延迟打开串口和退出时关闭。build_packet统一构造数据帧，parse_packet验证返回帧。read_status依次读取温度、位置和电压，move_servo发送位置及运行时间，set_motor_speed发送有符号速度。所有输入在编码前验证ID、位置、时间和速度范围，串口读取采用单调时钟限制超时。

JointCommandPlanner从YAML加载关节配置并支持J1、joint1_base_yaw、servo1等别名。位置模式关节根据当前与目标位置差值和分段运动规则计算run_time_ms；速度模式关节根据目标相对当前位置选择正负速度。JointCommandExecutor在速度模式到目标的过程中轮询位置，并在达到容差或超时时停止。

## 6.4 操作终端

Windows操作终端和Ubuntu独立终端均提供上下、水平摇杆、J5逆/顺时针、J6松/闭、抓紧、Home和全部停止。程序启动时读取J1至J4实际位置，或在dry-run中使用Home值。水平摇杆把屏幕坐标映射为机器人前后和左右速度，上下按钮映射为Z轴速度；每个80 ms控制周期根据雅可比计算下一组J1至J4位置命令。

J5和J6按钮使用按住运动、松开停止。J6抓紧为带颜色状态的锁定开关，锁定后以速度300运行并忽略松/闭误触，再次点击或全部停止后发送0速度。Ubuntu版本增加Linux串口自动发现和选择，优先枚举/dev/serial/by-id，再枚举ttyUSB和ttyACM设备，并检查读写权限。

## 6.5 模型后端适配器设计

ModelBackendManager维护CloudBackend和LocalBackend实例。两者实现相同的parse_task方法，接收用户指令、压缩后的当前图像、机器人状态、视觉状态和动作白名单，返回TaskPlan。管理器负责超时、重试、后端选择和日志，但不负责执行动作。TaskPlan进入SchemaValidator后逐字段检查task_type、object_id、steps、success_condition和risk_checks。

为了保持图像坐标一致，模型可使用缩放后的图像进行语义理解，但返回的bbox和center必须映射回600×480原始图像坐标。闭环控制器始终使用本地时间戳最新的原始坐标，避免模型处理延迟导致对旧帧执行动作。

## 6.6 任务状态机

任务执行器包含IDLE、OBSERVE、PLAN、ALIGN_GRASP、GRASP、VERIFY_GRASP、LIFT、ALIGN_PLACE、DESCEND_PLACE、RELEASE、VERIFY_PLACE、RETURN_HOME和ABORT状态。每个状态定义进入条件、允许动作、成功条件、超时和失败去向。ALIGN_GRASP和ALIGN_PLACE均为循环状态；GRASP和RELEASE均使用J6位置反馈；VERIFY_GRASP和VERIFY_PLACE使用连续多帧视觉条件。

## 6.8 日志与异常处理

任务日志至少记录任务ID、模型后端、模型调用时间、图像时间戳、对象ID、各状态进入退出时间、舵机位置、视觉误差、闭环迭代次数、停止原因和最终结果。异常分为输入异常、模型异常、视觉异常、规划异常、执行异常和硬件异常。任何异常首先停止持续速度命令，再根据反馈可靠性决定保持、退回安全高度或回Home。

串口超时、校验错误和权限错误必须向界面显示明确原因；视觉目标丢失不能被解释为“已完成”；模型返回未知动作直接拒绝；闭环超过最大迭代次数或3 s设计时间目标不等于立即判定失败，但应记录超标并按安全上限退出。日志和失败图像为第七章性能分析提供原始数据。

## 7.13 结果分析方法

完成实测后，应先检查数据完整性和异常值来源，再比较各指标与设计目标。若目标检测Precision达到95%但Recall较低，说明系统可能通过保守阈值减少误检却漏掉目标；若平均定位误差小于2 cm但最大误差较大，应分析工作区边缘、光照和相机畸变。闭环时间超过3 s时，需要区分图像处理延迟、模型调用延迟、舵机运行时间和稳定等待时间。任务总耗时超过60 s时，通过阶段日志定位主要瓶颈。

测试结论必须同时说明测试场景、样本数、重复次数和测量工具。对于未达标项目，应给出具体原因和改进方向，而不是删除失败数据。当前阶段可确认58项软件测试通过；其余性能结论需在完成真机标定、数据采集和整机测试后填写。

# 第八章 实现功能、作品特色与总结

## 8.1 当前已实现功能

项目已经实现总线舵机通信协议封装，可发送位置和Motor命令并读取温度、位置和电压；完成J1至J6的ID、范围、方向和机械尺寸配置；建立简化正运动学、TCP姿态和数值雅可比；完成Windows操作终端和可独立部署的Ubuntu 22.04终端；实现J1至J4笛卡尔小步控制、J5/J6速度控制、Home、全部停止和dry-run；完成ROS 2 URDF/Xacro、RViz显示及J6双指mimic映射；建立舵机稳定位置检测工具；完成58项软件测试。

目标检测主线、夹爪标记检测、视觉雅可比标定、云端/本地模型适配器、自动抓取状态机和放置闭环目前属于设计完成、待编码或待集成部分。报告不把Gemini RGB-D实验代码当作RGB主线已经完成的整机能力。

## 8.2 作品特色

第一，系统采用云端/本地双模型统一接口，使复杂语义能力和离线可用性可以按场景选择。第二，模型输出必须经过统一Schema、动作白名单和本地风险检查，安全边界不依赖模型自觉。第三，抓取、夹紧、抬升验证、放置对准、逐步下降、释放位置和释放结果形成多个连续闭环，成功条件建立在真实反馈上。第四，J6利用位置稳定性判断接触，在不增加力传感器的情况下提供低成本抓取反馈。第五，运动学、配置、总线协议、操作界面和ROS 2显示模块相互独立，便于替换相机、模型或控制平台。第六，Windows开发和Ubuntu部署两条路径并存，Ubuntu版本可在缺少PyPI网络时使用系统软件包部署。

## 8.3 局限与改进方向

当前相机内参、安装外参、HSV阈值和视觉雅可比尚未实测；600×480@60 fps需要确认相机驱动支持；本地模型具体型号、量化方式和推理延迟尚未确定；RGB单目条件下的叠放高度判断对光照、遮挡和物体尺寸敏感；J2负载误差、机械间隙和结构柔性尚未通过真机数据建模；J5/J6速度模式还需要更严格的超时和位置边界联锁。

后续应优先完成相机标定和RGB样本采集，实现橙色爪尖检测与目标跟踪；然后自动执行J1至J4小步扰动，建立不同工作姿态的视觉雅可比；再依次集成单目标对准、J6接触闭环、抬起验证、放置闭环和结果确认。完成全部真机测试后，将第七章待实测表替换为原始数据、统计图和现场图片。

## 8.4 总结

本作品提出了一套面向桌面抓取、放置和叠放任务的云端/本地双模型RGB视觉闭环机械臂方案。系统以结构化任务隔离模型与硬件，以运动学和视觉误差生成小步动作，以舵机位置和多帧图像验证抓取与放置结果。现有项目已完成底层通信、关节配置、运动学、操作终端、ROS 2显示和软件测试，为上层智能任务提供了可验证的执行基础。设计指标为目标检测准确率大于95%、夹爪和TCP定位误差小于2 cm、抓取与放置闭环收敛时间小于3 s、单次任务耗时小于60 s。后续工作重点是补齐视觉与模型模块、完成真机标定并用可追溯数据验证上述指标。

## B.1 总线协议校验和与数据帧

```python
HEADER = b"\x55\x55"

def checksum(body: bytes) -> int:
    return (~sum(body)) & 0xFF

def build_packet(servo_id: int, command: int, params: bytes = b"") -> bytes:
    length = len(params) + 3
    body = bytes((servo_id, length, command)) + params
    return HEADER + body + bytes((checksum(body),))
```
