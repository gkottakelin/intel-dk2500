### 2.4 Gemini Pro Plus Windows 调试代码

已开发第二阶段 Windows 调试代码，用于验证相机是否能被系统、SDK 或 OpenNI 示例识别。

已实现脚本：

| 文件 | 作用 |
|---|---|
| `project/src/gemini_windows/diagnose_environment.py` | 检查 Python、numpy、OpenCV、pyorbbecsdk 导入是否正常 |
| `project/src/gemini_windows/windows_usb_check.py` | 检查 Windows PnP/USB 层面是否识别相机 |
| `project/src/gemini_windows/list_devices.py` | 使用 pyorbbecsdk 枚举 Orbbec 设备 |
| `project/src/gemini_windows/read_intrinsics.py` | 尝试读取相机内参 |
| `project/src/gemini_windows/camera_stream_test.py` | 尝试 RGB-D 实时显示、点击测距和样本保存 |
| `project/src/gemini_windows/run_openni_sample.py` | 运行 Windows OpenNI 资料包中的官方示例 |
| `project/src/gemini_windows/opencv_uvc_color_test.py` | 使用 OpenCV 读取 Gemini Pro Plus 的 UVC 彩色画面 |

当前调试结论：

| 现象 | 结论 |
|---|---|
| Orbbec Viewer 可以看到彩色、深度、红外、点云 | 相机硬件和官方 Viewer 工作正常 |
| Windows PnP 能看到 `VID_2BC5` 相关设备 | Windows 系统层面能识别设备 |
| `pyorbbecsdk2` 能导入但枚举设备数量为 0 | 当前 Python SDK 路线暂不稳定 |
| OpenNI 示例中的深度、点云等可用 | Gemini Pro Plus 当前更适合先走 OpenNI 示例验证 |
| OpenNI `ColorReaderPoll.exe` 报错 | 该示例尝试走 OpenNI 彩色流，不适合当前设备组合 |
| UVC 彩色流可作为彩色图读取路线 | 彩色图应优先用 UVC/OpenCV 或 `ColorReaderUVC.exe` 路线 |

当前建议路线：

| 数据 | 推荐读取方式 |
|---|---|
| 深度图 | OpenNI `SENSOR_DEPTH` |
| 点云 | OpenNI `GeneratePointCloud` 或后续自行用内参换算 |
| 彩色图 | UVC/OpenCV 或 OpenNI 资料包中的 `ColorReaderUVC` |
| RGB-D 融合 | 后续做时间同步、坐标对齐和手眼标定 |

## 4. 下一阶段建议

### 阶段 A：Windows 下稳定拿到深度数据

目标是把“官方 OpenNI 示例能看到深度”推进到“项目 Python 或 C++ 封装能读取深度矩阵”。

建议任务：

1. 复现 `DepthReaderPoll.cpp` 的深度读取逻辑。
2. 输出深度图尺寸、像素格式、中心点深度。
3. 保存一帧深度矩阵为 `.npy` 或 `.png`。
4. 建立深度图点击测距工具。

### 阶段 B：彩色图和深度图融合

目标是实现物块识别和测距闭环。

建议任务：

1. 用 OpenCV/UVC 获取彩色图。
2. 用 OpenNI 获取深度图。
3. 记录时间戳并做近似同步。
4. 对彩色图中目标物块取 ROI。
5. 在 ROI 内统计有效深度，得到目标距离。

### 阶段 C：从图像坐标到机械臂坐标

目标是让机械臂知道目标物块在自己坐标系中的位置。

建议任务：

1. 读取或标定相机内参。
2. 使用深度值将像素点转换为相机坐标。
3. 做相机与机械臂基座的手眼标定。
4. 输出目标点在机械臂基座坐标系下的 `x, y, z`。

### 阶段 D：运动规划和舵机执行

目标是从目标位置生成关节命令并驱动机械臂运动。

建议任务：

1. 建立机械臂关节定义和限位配置。
2. 实现或接入逆运动学求解。
3. 将目标末端位姿转换为各舵机目标位置。
4. 调用 `bus_servo.py` 下发舵机动作。
5. 增加执行前安全检查、限位检查和急停策略。

### 阶段 E：ROS2/DK2500 迁移

目标是把 Windows 验证过的视觉、规划和控制逻辑迁移到 DK2500 上。

建议任务：

1. 在 DK2500 上安装 Ubuntu 和 ROS2。
2. 部署 Orbbec ROS2 wrapper。
3. 确认相机话题 `/color/image_raw`、`/depth/image_raw`、`/camera_info` 可用。
4. 把视觉定位、任务规划、舵机控制拆成 ROS2 节点。
5. 完成端到端抓取演示。

## 5. 主要风险

| 风险 | 影响 | 应对策略 |
|---|---|---|
| Python SDK 无法枚举 Gemini Pro Plus | Windows 下无法直接用 pyorbbecsdk2 获取 RGB-D | 先走 OpenNI 官方资料包和 UVC/OpenCV 路线 |
| Windows 和 DK2500 USB 环境不同 | 当前电脑验证结果不一定等同于 DK2500 | 把相机接入逻辑抽象成独立模块，迁移时只替换驱动层 |
| 彩色和深度流不同源 | RGB-D 对齐复杂 | 先做 ROI 深度估计，再逐步做 D2C 对齐 |
| 手眼标定误差 | 影响抓取精度 | 先用固定场景标定板和重复测量验证 |
| 舵机机械限位不明确 | 可能撞限位或卡死 | 在配置文件中加入每个关节的安全范围 |
| 虚拟环境不可移植 | 上传 venv 后其他机器难以使用 | 使用 `requirements.txt` 和环境重建说明替代上传 venv |
