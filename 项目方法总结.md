# 项目方法总结

> 本文总结项目中已经使用或确定采用的方法，按照“使用什么方法、解决什么问题、实现什么功能”的方式整理，便于后续写论文、答辩、报告和开发记录。

## 1. 项目工程化方法

| 使用的方法 | 解决的问题 | 实现的功能 |
|---|---|---|
| 项目目录分层管理 | 原始资料、代码、规划文档混在一起，不利于持续开发 | 建立 `project/` 作为开发目录，集中管理文档、代码、配置、测试 |
| 资料索引法 | 硬件资料、ROS2 资料、Windows SDK 资料分散，查找困难 | 建立项目资料索引，把 Gemini、DK2500、总线舵机资料并入目录 |
| 阶段化开发法 | 机械臂系统同时包含视觉、AI、规划、舵机，复杂度高 | 将系统拆成 Windows 验证、RGB-D 处理、机械臂控制、ROS2 迁移等阶段 |
| 模块化架构法 | 相机、AI、规划、舵机之间直接耦合会导致调试困难 | 将系统拆分为视觉模块、语言理解模块、运动规划模块、执行控制模块 |

## 2. 总线舵机控制方法

| 使用的方法 | 解决的问题 | 实现的功能 |
|---|---|---|
| UART 串口通信方法 | 上位机需要通过串口与舵机控制板通信 | 使用 COM 口和 `115200bps` 串口速率发送舵机指令 |
| 协议帧组包方法 | 舵机通信协议要求固定帧头、ID、长度、指令、参数和校验 | 在 Python 中生成符合手册要求的舵机控制帧 |
| 校验和校验方法 | 防止舵机接收错误帧或解析错误数据 | 实现协议校验和计算与返回帧校验 |
| 状态读取解析方法 | 需要读取舵机温度、角度、电压来判断运行状态 | 实现 `read_servo_status(com_port, servo_id)` |
| Servo 位置控制方法 | 需要让舵机运动到指定位置 | 实现 `set_servo_position(com_port, servo_id, target_position, run_time_ms)` |
| Motor 速度控制方法 | 需要让舵机按指定速度连续转动 | 实现 `set_motor_speed(com_port, servo_id, speed)` |
| 假串口单元测试方法 | 没有硬件时无法稳定测试协议代码 | 使用测试代码验证组帧、校验、状态解析等逻辑 |

## 3. Gemini Pro Plus 相机接入方法

| 使用的方法 | 解决的问题 | 实现的功能 |
|---|---|---|
| Orbbec Viewer 观察法 | 先确认相机硬件是否能正常输出图像 | 在上位机观察彩色、深度、红外、点云数据 |
| Windows PnP/USB 检查法 | 判断问题出在系统识别层还是 SDK 枚举层 | 通过 `windows_usb_check.py` 查看 Windows 是否识别相机 |
| SDK 导入诊断法 | 判断 Python 环境是否正确安装依赖 | 通过 `diagnose_environment.py` 检查 numpy、OpenCV、pyorbbecsdk |
| pyorbbecsdk 枚举法 | 尝试使用 Python SDK 直接访问相机 | 通过 `list_devices.py` 验证 SDK 是否能发现设备 |
| 本地资料溯源法 | Python SDK 不能枚举设备，需要寻找可用读取路线 | 在 `gemini深度相机windows资料` 中定位 OpenNI 示例和源码 |
| OpenNI 深度读取方法 | 需要稳定获取 Gemini Pro Plus 深度图 | 使用 OpenNI 示例验证 `SENSOR_DEPTH` 深度流 |
| UVC 彩色读取方法 | OpenNI 彩色流示例不适合当前设备组合 | 使用 UVC/OpenCV 或 `ColorReaderUVC` 读取彩色图 |
| OpenNI 示例封装方法 | 每次手动进入资料目录运行示例不方便 | 实现 `run_openni_sample.py` 统一启动官方示例 |

## 4. RGB-D 图像处理方法

| 使用的方法 | 解决的问题 | 实现的功能 |
|---|---|---|
| 深度图测距方法 | 需要知道图像中某个点到相机的距离 | 在深度图中读取目标像素或 ROI 的深度值 |
| ROI 深度统计方法 | 单个深度像素可能有空洞或噪声 | 对目标区域内有效深度取中位数或均值，提高测距稳定性 |
| 彩色图目标识别方法 | 需要在环境中找到物块位置 | 后续使用颜色阈值、轮廓检测或目标检测模型识别物块 |
| 相机内参反投影方法 | 需要把像素坐标和深度转换为三维坐标 | 使用 `X = Z * (u - cx) / fx`、`Y = Z * (v - cy) / fy`、`Z = depth` |
| 点云生成方法 | 需要把深度图转换为空间点集合 | 参考 OpenNI `GeneratePointCloud` 示例生成点云 |
| RGB-D 对齐方法 | 彩色图和深度图分辨率、坐标系可能不同 | 后续进行 D2C 对齐或通过标定建立映射关系 |

## 5. AI 自然语言与运动规划方法

| 使用的方法 | 解决的问题 | 实现的功能 |
|---|---|---|
| 自然语言结构化方法 | 人类输入是自然语言，机械臂需要明确动作参数 | 将“抓取红色物块”转换为目标类别、位置、动作类型 |
| 任务分解方法 | 一个自然语言命令可能包含识别、移动、抓取、放置多个步骤 | 把任务拆解为感知、定位、规划、执行、确认 |
| 坐标转换方法 | 相机看到的是相机坐标，机械臂执行需要基座坐标 | 通过手眼标定得到相机坐标到机械臂坐标的变换 |
| 逆运动学方法 | 目标点是末端空间位置，舵机需要关节角度 | 将末端位姿求解为各关节目标角 |
| 安全限位方法 | 机械臂可能撞限位、撞桌面或自碰撞 | 后续在配置中限制每个关节角度、速度和工作空间 |
| 执行闭环方法 | 舵机执行后需要确认状态 | 通过状态读取接口检查温度、电压、角度，判断动作是否完成 |

## 6. ROS2/DK2500 迁移方法

| 使用的方法 | 解决的问题 | 实现的功能 |
|---|---|---|
| Windows 先验证、Linux 后迁移 | 当前开发电脑和 DK2500 环境不同，直接迁移风险高 | 先在 Windows 验证相机和舵机，再迁移到 DK2500 |
| ROS2 节点化方法 | 多模块系统需要统一通信和启动管理 | 将相机、识别、规划、舵机控制拆为 ROS2 节点 |
| ROS2 Topic 通信方法 | 视觉数据、目标位置、控制命令需要解耦传输 | 使用图像话题、目标话题、关节命令话题组织数据流 |
| ROS2 Launch 启动方法 | 多节点系统手动启动复杂 | 使用 launch 文件统一启动相机、识别、规划、控制节点 |
| 配置文件参数化方法 | 不同 USB、舵机 ID、关节限位、相机参数需要可修改 | 使用配置文件保存硬件参数，避免写死在代码中 |

## 7. 环境复现方法

| 使用的方法 | 解决的问题 | 实现的功能 |
|---|---|---|
| 依赖清单替代虚拟环境上传 | `.venv-gemini` 体积大、包含绝对路径、不可跨机器稳定使用 | 使用 `requirements.txt` 和环境重建命令复现开发环境 |
| 版本快照方法 | 需要记录当前虚拟环境中已经安装过的包 | 使用 `requirements-venv-gemini.txt` 保存当前观察到的包版本 |
| 外部资料包说明方法 | OpenNI 和 Orbbec Viewer 资料包可能较大或受许可约束 | 在文档中记录本地资料包路径，不默认上传大体积二进制资料 |
